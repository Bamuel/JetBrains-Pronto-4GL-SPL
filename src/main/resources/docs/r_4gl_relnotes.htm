    • If the program is specified with a full path name then the leading directories will be ignored when generating the operations table.

Release 7.7v0.16

  • SELECT FOR-UPDATE NO-WAIT

    The NO-WAIT clause has been added to the SELECT statement. If specified it must immediately follow the FOR-UPDATE clause (i.e. FOR-UPDATE NO-WAIT).

    This clause specifies to not wait for locked rows and to abort the SELECT statement if a locked row is encountered. If an ON ERROR clause is present then this will be invoked if a lock is encountered.

    The user will not be given the option to retry the lock request if a lock is encountered.

    The application cannot commit transactions during the SELECT loop. Doing so will cause the SELECT loop to fail.

    A SELECT that specifies FOR-UPDATE NO-WAIT is more efficient as it does not need to cater for lock retries and application transaction commits during the loop.

Release 7.7v0.14

  • REFRESH SECTIONS

    • The REFRESH SECTIONS statement has been added.

    • This will cause all sections within the section group to be refreshed.

    • If called from a child section, the controlling parent will also have its data refreshed (that is, a REFRESH DATA).

  • EVENT-SECTION

    • The EVENT-SECTION control group has been added for a SCREEN.

    • This section is invoked when a trappable event within the screen has occurred, such as clicking an OLE control or receiving an input timeout.

    • The EVENT-SECTION should be places immediately after the DETAIL section.

      The syntax is:

      EVENT-SECTION

      [TIMEOUT [IS] <expression]

      <procedural-code>

      END-EVENT-SECTION

    • Events can trigger the EVENT-SECTION when the screen is at mode selection or when accepting input in the Detail section.

    • The EVENT-SECTION code is processed in a show mode when triggered while at mode selection. Any Accept operation processed will only display the current value.

    • When triggered while accepting input in the Detail section the EVENT-SECTION code will run using the current screen mode.

    • Unless the section processes an EXIT, BREAK, CONTINUE or RE-ENTER statement the application will return to the point that it was at when the event occurred (i.e. an Accept or mode selection).

      EXIT will exit the screen.

      BREAK/CONTINUE will either return to mode selection or work the same as they would if encountered normally in the Detail section.

      RE-ENTER will return processing to that Accept, or return to mode selection if not processing the Detail section.

    • The TIMEOUT clause implements a timer. This timeout event occurs both when the screen is at mode selection and when waiting for input within the Detail section. If an Accept also has a timeout specified then this takes precedence. The timeout expression is evaluated prior to each input operation within the screen so that it can dynamically vary (e.g. be set to 0 to disable).

    • The existing OLE-GET-EVENT(ole-control-id,event-id) function is used to get OLE event details. The following will return the most recent OLE event values sent by the Web Client.

    • If called with an ole-control-id of 0 it will return the ole-control-id sent by the Web Client. A 0 will be returned if processing a timeout event.

    • If called with an ole-control-id matching the one sent by the Web Client and an event-id of 0 it will return the event-id sent.

    • If called with the ole-control-id and event-id matching the ones sent by the Web Client it will return the event notification ID sent by the Web Client.

    • The OLE-GET-PROPERTY function can be used to return the associated value for the event notification. The interface ID specified should be either 0 or the notification ID value returned by OLE-GET-EVENT.

  • SCREEN-STYLE clause added

    • The SCREEN-STYLE clause has been added to a SCREEN declaration.

    • This specifies a numeric screen style which will be sent to the Web Client.

    • The syntax is:

      SCREEN-STYLE [IS] <numeric-expression>

  • Miscellaneous

    • The report viewer now supports viewing non-xml reports with line widths up to 5000 characters.

Release 7.7v0.11-13

  • MODE-WHEN clause added

    • The MODE-WHEN clause has been added to the MODE declaration. This specifies a condition that determines if this mode is valid for the screen. This condition is evaluated once only when the screen is invoked.

    • If both MODE-WHEN and WHEN conditions are present, then the MODE-WHEN will indicate if the mode is valid for the screen, and the WHEN condition will indicate if the mode is valid for the current record.

    • If a MODE-WHEN condition is not specified then the legacy behaviour of the WHEN condition is used when a screen is invoked. If there is no MODE-WHEN and the WHEN condition is false when the screen is invoked then the mode will not be valid for the screen unless a SHOW-ALWAYS clause is also specified.

  • LOCAL MODE declaration

    • The LOCAL MODE clause has been added to the SCREEN declaration.

    • A local mode is the same as a standard global mode, but can only be referenced by the screen that it is defined in.

    • A local mode may refer to local and parameter fields in any conditions or expressions. The local/parameter fields must be defined prior to referencing them.

    • Local modes will be automatically added to the allowed mode list for the screen if they are not explicitly listed.

    • ALWAYS-SHOW will be assumed for a local mode unless a MODE-WHEN clause is specified.

    • Any conflicting screen clauses (e.g. WHEN) must not be specified immediately after local mode definitions otherwise they will be assumed to be part of the mode definition.

    • A local mode may not be passed in the mode list of the DO statement.

    • For example:

      • screen my-screen

      • parameter is p1 pic x

      • local mode local-correct correct process mode-when p1 = 1

      • allowed search local-correct

  • MODE declaration

    • The MODE and LOCAL MODE declarations may now specify multiple mode definitions.

    • For example:

      • mode mode1 correct process

      • mode2 entry prompt "entry"

      • mode3 prompt "Do stuff"

    • Internal GET LOCK by unique key optimisation.

Release 7.7v0.5

  • DEFAULT-IF-WHEN-FALSE clause for ACCEPT

    • The DEFAULT-IF-WHEN-FALSE clause has been added to the ACCEPT statement.

    • Traditionally if an ACCEPT has a WHEN clause that evaluates to FALSE then in Entry mode the field's value will not be set to the DEFAULT value if one was specified. This was to cater for multiple overlaid fields in legacy applications. The DEFAULT-IF-WHEN-FALSE clause overrides this behaviour and will set the field to the default value in Entry mode even if the WHEN condition evaluates to FALSE.

    • This clause is only valid in FORM-ENTRY and RESPONSIVE screens.

Release 7.7v0.3

  • CALL-URL

    • The ALLOW-REDIRECTION clause has been added. This flags that URL redirection or forwarding should be allowed.

    • Added SFTP protocol support. This can be specified as the protocol within the URL. The URL also specifies the path to the remote file.

      (e.g. "sftp://remotehost/mydir/myfile");

      A HTTP-METHOD of "upload" must be specified to send a file.

    • The SFTP-COMMAND clause has been added. This specifies SFTP commands to run prior to the file transfer. Multiple SFTP-COMMAND clauses can be specified. The commands supported are those defined by libcurl. Refer to the online help for CURLOPT_QUOTE for details of the command supported.

      The syntax is:

      SFTP-COMMAND [IS] <command-string>

    • The SFTP-POST-COMMAND clause has been added. This specified SFTP commands to run after the file transfer. Multiple SFTP-POST-COMMAND clauses can be specified.

      The syntax is:

      SFTP-POST-COMMAND [IS] <command-string>

    • The SFTP-CREATE-DIRS clause has been added. This specifies that directories in the SFTP file path should be created if not present.

    • The URL-USER clause has been added. This allows for the specification of the user and password to be used for the connection.

      The format is:

      URL-USER [IS] <user-name> [<password>]

    • The URL-LOGIN-OPTIONS clause has been added. This allows a login options string to be specified in accordance with RFC 2384 and RFC 5092.

      The syntax is:

      URL-LOGIN-OPTIONS <options-string>

  • CHECK-USER-AUTH

    • The CHECK-USER_AUTH function has been extended to allow a license group number or a user name to be specified as the first parameter.

    • The new usage is:

      CHECK-USER-AUTH (<user-name>|<group-number>, <product-IDs>)

  • Miscellaneous

    • Added support for hide/unhide data grid columns.

Release 7.7v0.2

  • CALL-URL FORM DATA

    • The FORM clause has been added to the CALL-URL statement. This specifies form data to be posted as the content/body of the call.

      The syntax is:

      FORM NAME [IS] <name>

      DATA [IS] <text-data>] | [FILE [IS] <file-path>

      [TYPE [IS] <data-type-name>]

    • Multiple FORM clauses can be specified for a multi-part form.

    • NAME is the name of the form item.

    • DATA specifies the content as a text field.

    • FILE specifies a the path of a text file that contains the content.

    • TYPE is optional and specifies the type of form data.

    • If form data is specified then a HTTP-BODY cannot also be specified.

  • CALL-URL HTTP-HEADER

    • Multiple HTTP-HEADER clauses can now be specified. Each will be added to the header and separated by CR/LF.

Release 7.7v0.1

  • Data-grid row marking/selection

    • The ROW-SELECTION-OBJECT clause has been added to the screen declaration section for a DATA-GRID screen. This nominates an object of type memory to record the rows selected by the user. The key for index 1 of the object must specify the fields to uniquely identify selected rows.

      The syntax is:

      ROW-SELECTION-OBJECT [IS] <memory-object-name>

    • When a row is selected a record is added to the nominated object. The fields of the object will be set to the current values of the row selected.

    • When a row is de-selected the record with the corresponding index 1 key for that row will be deleted.

    • Any existing content of the row selection object will be cleared when the screen begins.

    • The ROW-SELECTION-COUNT() function has been added. This returns the number of rows selected for the current screen.

    • The ROW-SELECTION-CHANGED() function has been added. This returns a non-zero value when the rows are selected or de-selected. This can be used to determine when to refresh statistics of the rows selected.

  • CHECK-USER-AUTH function added

    • The CHECK-USER-AUTH (<user>, <product-IDs>) function has been added.

    • This returns a non-zero value if the license group of the specified user is licensed for one or more of the product IDs listed. A value of zero returned otherwise.

Release 7.4v0.8

  • OFFSET and FETCH clauses for SELECT

    • The OFFSET and FETCH clauses have been added to the SELECT statement.

    • The OFFSET clause specifies the number of rows to offset/skip at the start of the result set.

    • The FETCH clause specifies the maximum number of rows to return from the result set.

    • If used, these clauses must be specified immediately after the ORDER BY clause for the query. An ORDER BY clause must be specified.

    • The syntax is:

      ORDER BY <field-list> [ OFFSET <expr> [ROW[S]] ]

                     [ FETCH [FIRST|NEXT] [<expr>] [ROW[S]] [ONLY] ]

    • The FIRST and NEXT keywords are optional and ignored if present. If <expr> is not specified for FETCH then 1 row is returned.

    • The value for <expr> can be any numeric expression. This expression is evaluated separately to the SELECT statement.

      For example:

      select * from deb-master order by accountcode offset 5 rows fetch 10 rows only

      select * from deb-master order by accountcode fetch first :lf-rows rows only

      select * from deb-master order by accountcode fetch first row only

      These clauses are only valid for non-screen based SELECT statements. That is, they cannot be used in the Select statement for a screen.

      Windows implementations will require SQL Server 2012 or later to use these clauses.

  • AS alias clause for SELECT

    • The AS keyword can be specified in the FROM clause to define an alias for the table/object. The table will be known by this alias within the query, and references to the actual object name will be invalid.

    • The syntax is: FROM <object-name> [AS <alias-name>] ...

      For example:

      select dm.accountcode, dm.balance, dt.tr-amount

      from deb-master as dm

      inner join deb-trans as dt on dm.accountcode = dt.accountcode

      where dm.balance > 1000

    • For consistency, the AS keyword is also valid when selecting columns. This will be considered the equivalent of using the = assignment for a calculation. Specifying "select balance as b" is equivalent to "select b = balance". The field specified after the AS must exist, and is processed as a calculation rather than an alias.

Release 7.4v0.4/5

  • Database Sequences

    • Support has been added for database sequences.

    • A sequence is defined as a Pronto object of type sequence-number. The name of the sequence is defined by the file name for the object, in the same way that a normal database table is specified. A sequence-number object must consist of a single numeric field. No indexes may be defined. By default, the sequence will start at 1 and have a maximum allowed value based on the number of integer places defined for the sequence field.

    • Operations on a sequence are controlled by the standard I/O statements for a Pronto object.

    • To create a sequence you perform an OPEN CREATE or OPEN TRUNCATE.

    • To remove a sequence you perform a CLOSE AND REMOVE.

    • To get the next value for the sequence you issue a GET NEXT or GET operation. The single numeric field defined for the sequence object will be set to the sequence value. A GET CURRENT is implemented for consistency, however it does not look up the sequence. It simply returns the last value set for the sequence field within the program.

    • If an application issues two consecutive GET (NEXT) operations for a sequence without any other RDBMS I/O operation between them, an end of file error will be returned.

    • A GET LAST operation will return the end of the system wide sequence number stream. This number will not have been issued to an application yet. If the sequence uses caching there may be a gap of more than one between this number and the next to be allocated to an application. Issuing a GET LAST will not issue a new sequence number, and will have no effect on the sequence number stream. This operation is intended for sequence administration purposes, and must not be used to assign sequence numbers within an application.

    • An UPDATE or INSERT operation can be used to set/restart the sequence. The next sequence number to be allocated will start at the specified value.

    • The PROCOPY utility can be used to import/export or copy the sequence, in exactly the same way as any other table. A sequence will be seen as a single column single row table.

    • Database sequence numbers do not form part of any transaction, and cannot be rolled back once allocated.

    • Database sequence numbers will be unique and in order, but are not guaranteed to be contiguous.

    • Windows implementations will require SQL Server 2012 or later to use database sequences.

      The syntax is:

      <object1> INNER JOIN <object2> ON <condition>

    • The join types are:

      INNER JOIN - A tradition join where only rows that satisfy the join condition will be returned.

      LEFT OUTER JOIN - All valid rows from the left objects, plus qualifying rows from the right objects. When there is no qualifying row from the right object, then NULL values will be returned for each of its selected columns.

      RIGHT OUTER JOIN - Same as LEFT but the roles of the two objects is reversed.

  • SELECT statement

    • Where possible the full SQL code for the SELECT statement will be issued directly to the database. The runtime will determine if this is possible. If not possible, the SELECT statement will be processed internally by the 4GL runtime in the traditional manner.

    • Issuing the full SQL code for the query to the database directly can result in major improvements to performance for SELECT statements that specify multiple objects, have a large degree of filtering specified in the WHERE clause, or specify aggregate functions. This is because it reduces the amount of data that needs to be retrieved by the runtime in order to process the query itself.

    • This currently only applies to the SELECT statement (that is, SELECT loop) and not to the SELECT clause of a SCREEN.

    • The DATABASE-SQL or LOCAL-SQL clause can be specified directly after the query to force where the query will be processed. If neither is specified, then the runtime will determine the method of processing.

    • This only applies to queries that reference objects of type INDEXED or DB-SEQUENTIAL.

    • If the statement specifies functions that are not directly supported by the databases, then the traditional SELECT processing must be used.

      The supported functions are:

      ABS, AVG, COUNT, DAY, LOWERCASE, LTRIM, MAX, MIN, MONTH, RTRIM, SQL-SUBSTRING, STRLEN, SUM, TODAY, UPPERCASE, YEAR.

    • If FOR-UPDATE is specified then the direct SQL code will only be used if all fields for the object are selected (with an *). The LOCK-METHOD must also be set to NO-WAIT, as there is no ability to retry a locked record when the database is in full control of the SELECT processing. If the LOCK-METHOD is not NO-WAIT then traditional processing will be used for the SELECT.

    • All expression processing is passed to the database and not performed by the runtime. This means that it is possible to have database specific differences in how some expressions are evaluated. In particular, you should avoid fractional arithmetic (that is division) in the WHERE cause as differences in precision and rounding can cause inconsistent comparison results.

    • The databases do not have the concept of a zero date or time, so any comparisons of date and time fields to a literal value of zero should be avoided. You can however set a host date or time field to zero and compare to that.

    • A SELECT statement directly issued to the database cannot be restarted part way though. If the program performs a CLOSE ALL this will cause all direct SELECT statements to be closed. The SELECT loop will fail at the next iteration of the loop after a CLOSE ALL.

    • The SQL-UPDATE statement can be used to update multiple rows based on the conditions specified.

    • The syntax is:

      SQL-UPDATE [ALL] <object>

    • Fields that are not of TYPE ALPHA are now stored differently in text files.

    • Previously numeric, date or time type fields were not written as text and were written as their internal binary values.

    • The external text representation of the field is now stored (and read).

    • This can cause compatibility issues with any code that relied on the binary data for the field being written to the file.

  • Screen Context Sensitive Help

    • The HELP-CONTEXT clause has been added to the SCREEN/PROCEDURE/MENU declaration. This allows you to specify screen specific context sensitive help. When in this screen this help takes precedence over help specified using the SET-HELP-CONTEXT function.

    • The syntax is :

      HELP-CONTEXT [IS] <help-file-expr> <context-id-expr> name of the help file. <context-id-expr> is a numeric expression or literal that specifies the context ID.

  • Component Library Definition file

    • The 4GL compiler cannot generate an operations table for a program that references routines in a component library that it cannot find or has not been compiled. This can cause difficulties when attempting to compile component libraries that cross reference each other.

    • To help overcome this the compiler will instead generate a component library definition file (.cld) for the compilation of any component library that failed purely because of unresolved external components.

    • The .cld file can be used instead of an operations table when the compiler is attempting to resolve component library references.

    • Once all the required component libraries have been compiled, you will have either an operations table or a component library definition file for each one. You can then go back and recompile any component libraries that only have the component library definitions file in order to generate an operations table.

    • Once an operations table has been successfully generated any component library definitions file for that component library will be removed.

  • Audit Log

    • All audit information is now written to a database table rather than a text file. A single row is written to the table for each audited operation.

    • A separate audit log table is stored for each database or schema used and will store all audit information for Pronto Xi tables in that database or schema.

    • The TO clause of the AUDIT statement is now obsolete and will be ignored if specified.

    • The mode line has now been separated from the standard screen area. This allows for the mode area to be moved.

    • This feature has been implemented in a way that maintains compatibility with existing applications.

    • The mode area is associated with the window for a screen. When a screen inherits the window of a parent screen the use of a mode area is solely determined by the parent screen that owns the window.

    • Modes will be displayed as traditional buttons and the mode area not separated in the following situations:

      • The screen specifies the location of modes using PROMPT @<row>,<col>.

      • The screen has a single mode not requiring mode selection or the OK and Cancel buttons to be displayed.

      • Only windows for screens will create a mode area. Windows for procedures or menus will not create mode areas.

      • The screen inherits a parent window that does not have a mode area.

  • Dictionary Table Name Suffix

    • To avoid conflicts with the table names of dictionaries when multiple versions of the same dictionary are required to be stored in the RDBMS, a suffix will be automatically attached to the dictionary table names. The suffix is taken from the ${DICT}-dict.ver file if present for the dictionary. This applies to all dictionaries (i.e. PRODICT, PRODISTDICT and PROUSRDICT).

    • The suffix information is taken from the first line of the -dict.ver file. This is expected to be the dictionary version number. If a different suffix is required this can be specified in a second pipe (|) separated field on that line. For example, to use test as the suffix the line would be: 700.1|test.

    • If a fullstop ('.') is specified this will automatically be substituted for a '_'. You should ensure that only characters that are valid within an RDBMS table name are specified (alphas and numerics only are recommended).

    • If a value of NO_SUFFIX is specified, no suffix will be attached.

  • Dictionary Distribution

    • Dictionaries are now distributed using comma separated flat files with a .dif extension. Previously .dat files were used.

    • The -i (import) option of the dictload utility now loads from .dif files.

    • A -e (export) option has been added to export the dictionary in the .dif distribution format. This requires the specification of the directory where the distribution version is to be written to directly after -e. For example, prospl dictload -e $SRC -d $MASTERDICT

  • Sysinfo Mapping

    • Many shared tables are defined as existing in $Pronto Xi/sysinfo. The standard file name to table mapping mechanism used by PRONTO will result in all tables in this area being contained in a schema/database named 'sysinfo'. This occurs regardless of what the $Pronto Xi environment is set to. Because of this it has traditionally not been possible for RDBMS implementations to have multiple sysinfo areas within the same database instance.

    • A sysinfo mapping mechanism has been added to allow tables that are defined as being in $Pronto Xi/sysinfo to be associated with a different schema or database depending on the current directory that the application is running in.

    • The runtime will look for entries in a table named db_sysinfo in a schema/database named prontosys. The column named "data_area" contains the schema/database name of the data area whose sysinfo area to be mapped. The column named sysinfo_name contains the schema or database name to be used instead of sysinfo.

    • If the db_sysinfo table does not exist, or there is no entry for the current data area in the table, then the sysinfo area will not be mapped.

    • The 4GL runtime is not responsible for the creation or maintenance of the db_sysinfo table. This is the responsibility of the application software. This includes ensuring that the table has the correct access permissions.

    • Other columns can be defined in the db_sysinfo table if needed. The runtime only requires the data_area and sysinfo_name columns to be defined.

  • Dockable Web Windows

    • An application can launch a dockable HTML window that forms part of the client process. These windows can also be floated outside of the normal client area. Care should be taken to avoid Web pages that request keyboard focus as these might cause keyboard focus conflicts with the Pronto Xi screens or menu.

    • The SET-WEB-WINDOW (<tag>, <title>, <URL>, <flags>) function will launch a new Web window.

      The parameters are:

      <tag> - Alpha tag used to identify this window.

      <title> - Text string to be displayed as the windows title.

      <URL> - URL of the web page or HTML document to be used.

      <flags> - Numeric flag values:

      1 - Dock at the left when created.

      2 - Dock at the right when created.

      3 - Dock at the top when created.

      4 - Dock at the bottom when created.

      8 - Force docking location (otherwise uses last location.

      16 - Set Auto Hide mode for window.

      If multiple flag settings are required, the values should be added together. However, only one docking location value can be specified. If no docking location is specified, the default will be to float the window. Unless the force docking location flag is set, the previous saved docking location for this window will be applied.

    • If a window has already been created with the specified <tag> value, that window will be reused.

    • To delete a window, specify the <tag> of the window with a blank <URL> value.

    • By default when traditional #P or standard XML reports are generated there will be additional padding bytes (spaces) added to maintain alignment if UTF-8 characters are printed.
