CALL statement

Executes the specified 4GL program.

The operations table file (.op7), the compiled form of the .spl program file, must be present.

For more information on how to execute non-4GL programs, see COMMAND statement.

Syntax

call program-name
   [parameter[s] [is|are] alpha-expression ...]
   [max-parameters [are] numeric-expression]
   [returning field-list]
   [home [is] alpha-expression]
   [batch]
   [leave-files-open]
   [external]
   [background|no-wait|wait]
   [on error 
      procedural-statements
    [else 
      procedural-statements]
    end-on]

Clauses

Clause Description

program-name

The name of the 4GL program to execute. It can be any valid alphanumeric expression. If a full path name is not specified the program is searched for in the directories listed by the PROPATH environment.

PARAMETERS

A list of parameters to pass to the program. Only alphanumeric parameters can be passed. Arrays are supported.

MAX-PARAMETERS

Allows an expression to be specified that evaluates to the maximum number of parameters to be passed, regardless of the number of parameters specified.

RETURNING

The RETURNING clause allows the receipt of return values from child 4GL programs. If the initial routine of the child program contains the RETURNING clause, the contents of these fields is passed back to the fields specified in the CALL statement. A runtime error occurs if insufficient return values are passed back to the calling program.

The RETURNING clause in the CALL statement does not need to specify fields for all return values.

Arrays are supported.

No-join fields can be specified (that is, a field prefixed with a no-join object name).

HOME

Execute the command with this home directory.

BATCH

Submit this command to queue zero of the Pronto Xi batch queue. The program will not wait for the command to be executed.

LEAVE-FILES-OPEN

Specifies that the files that are currently open in the calling program are not to be closed when the child program is called. This also means that record locks are maintained.

This is accomplished by using a new 4GL interpreter to run the child program, instead of loading the child program into the current interpreter. This can impose a greater overhead on system resources (for example, memory and process slots) as well as possibly slowing down the load time of the child program.

Any records that are locked by the calling program cannot be locked by the child program.

EXTERNAL

Opens a new 4GL interpreter to run the child program. This is different from the LEAVE-FILES-OPEN clause, as all files are closed beforehand.

BACKGROUND or NO-WAIT

Flags that the child program should be detached from the calling program and run in the background on graphical systems (for example, Windows and X-Windows). The calling program will return immediately and will not wait for the child to complete. The child program will run independently of the calling program.

The RETURNING clause cannot be used as this relies on the calling program waiting for the child program to complete. The EXIT-STATUS function will only indicate the success of the command and not the actual exit status of the child program.

This clause only affects Windows based systems.

WAIT

Wait for child. This is the default option.

ON ERROR

Perform these statements if the specified command could not be executed. If the ELSE clause is included, the procedural statements following the ELSE are performed if the operation is successful.

The ELSE section of an ON ERROR is only performed if there is no error.

Notes

  • The PARAMETERS clause of the CALL/QUERY/COMMAND statements allow numeric values to be specified. Parameters for external programs are still passed in alphanumeric format; however, numeric values are automatically converted into their preferred alphanumeric format when passed.

  • If the initial routine of the child program contains the PARAMETERS clause, the parameter values passed are automatically moved into these fields (with the appropriate data conversions performed where necessary). A runtime error occurs if insufficient parameters are passed to the child program. The PARAMETERS clause does not need to define all expected parameters.
