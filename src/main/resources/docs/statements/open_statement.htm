OPEN statement

Opens the physical file associated with an object.

If this statement is not issued for an object, the file is opened automatically the first time it is referenced using the default file (table) name defined. If the object was previously opened with a different file name, this one is used again if an automatic open is performed.

Syntax

open object-name
   [file [is] file-name]
   [lock]
   [create]
   [truncate]
   [permanent|temporary]
   [private]
   [no-triggers]
   [local]
   [read-only]
   [index-on-close]
   [on error [error-number ...]
      procedural-statements
    [else
      procedural-statements]
    end-on]

Clauses

Clause Description

FILE

Open this physical file instead of the current default file name. file-name can be any valid alphanumeric expression.

The initial default file name is specified in the objectâ€™s definition. The file name specified in the FILE IS clause becomes the new default file name after the OPEN has been performed.

LOCK

Open the file for exclusive use. This clause is not supported on all file types and database implementations.

CREATE

Create the physical file / table for the object if it does not already exist.

OPEN CREATE is not supported within transaction control. The runtime must switch the default database / schema to which it is connected in order to create the table. This switch can't be done with an active transaction or with any active queries.

TRUNCATE

Reset the physical file / table to zero records if the file exists, otherwise create it. In the case of an IBM Informix database, if the table exists it is removed and recreated.

PERMANENT

TEMPORARY

If the TEMPORARY clause is included in the OPEN statement for an object, the file for this object is automatically removed when the program terminates (if the file still exists).

The TEMPORARY clause also implies TRUNCATE when opened.

If no explicit file name is specified using the FILE clause in the OPEN statement, a unique temporary file name is automatically generated. The default/current file name for the object will never be used unless the object is already flagged as temporary, in which case it will reuse its current temporary file name.

You can OPEN and CLOSE an object with a temporary status after its initial creation without changing its status or having the file automatically removed. However, if you specify the FILE IS clause when you OPEN the object, the temporary file is removed at this point (even if the file name passed is the same as the current temporary file).

Only a CLOSE with the AND REMOVE clause or an OPEN with the FILE IS clause can remove the temporary status of the object. Both these operations will cause the current temporary file to be removed.

Creating an object during a transaction will commit the transaction.

PRIVATE

This clause when used in conjunction with any of the file creation clauses (that is, CREATE, TRUNCATE and TEMPORARY) indicates that the file created should not allow read/write access by other users. Only the user who created the file will have any access to it.

NO-TRIGGERS

This clause can be used to disable a trigger for an object. Programs that perform data conversion, and so on, might need to turn trigger processing off for certain objects.

open deb-master no-triggers

This clause is specific to that object in that program or component. That is, if you open it with a different file name it has no impact on any other program or component. Therefore, child programs or any component libraries that might be run do not inherit this object specific setting.

LOCAL

This option is obsolete and should not be used.

This clause indicates that the file is located on the local machine instead of the server machine. A similar effect can be achieved by prefixing the name of the file with a colon (:) or exclamation mark (!).

The following file name indicates that the file is located on the local machine:

:/tmp/abcd

Both these options can be used on non-network versions without error.

READ-ONLY

Only allow read access of the file in the program. If this clause is not included, the file is opened for reading and writing.

INDEX-ON-CLOSE

Create all indexes to the table when the object is closed (instead of when the table is created). This only applies if the file was created/truncated as a result of this OPEN.

This can improve performance when a large number of records are inserted into this table by the program. The inserted records can only be accessed once the object has been closed and all indexes have been created.

If duplicate records are inserted on an index that does not allow them, the CLOSE operation will fail (with a duplicate record error). The indexes are not created if this occurs, and the program should take the necessary steps (for example, remove the table and create it again without this clause).

ON ERROR

Procedural statements to perform if the OPEN fails.

If the ELSE clause is included, the procedural statements following the ELSE are performed if the OPEN succeeds.

The ON ERROR clause of any I/O operation can have a list of error numbers specified (directly after the ON ERROR). The ON ERROR section is only performed if the error is one of those listed. If the error is not in the list, the program aborts with a full error detail screen.

NO-AUDIT

Turn off auditing of the object if auditing has been specified for the object in the dictionary.

Notes

  • Any OPEN statement that uses a FILE IS clause to identify the name of the file to create, will default to TEMPORARY unless the PERMANENT clause is used.

  • If the FILE IS clause is not used, PERMANENT is assumed.

  • Opening a flat file requires that file-system permissions (UNIX/WINDOWS) and the 4GL requested mode must match. By default, the 4GL will attempt to OPEN a file in READ-WRITE mode. If the file-system says the file is READ-ONLY, you must specify the READ-ONLY clause or the file will not open.
