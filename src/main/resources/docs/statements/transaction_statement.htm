TRANSACTION statement

Enables transaction control, or commits or rolls back the operations since transaction control was enabled.

The TRANSACTION statement can be used to mark the start and end of a logical transaction. A logical transaction is a set of file operations that must either all succeed, or have no effect if unsuccessful. This is done to ensure that the logical integrity of the data remains intact should a failure occur.

Only one transaction can be defined/active at a time.

Syntax

transaction begin [on object [, object ...]]
   [leaving object [, object ...]]
   [free-locks]
   [on error [error-number ...]
      procedural-statements
    else 
      procedural-statements
    end-on]

transaction commit 
   [on error [error-number ...]
      procedural-statements
    else 
      procedural-statements
    end-on]

transaction rollback 
   [on error [error-number ...]
      procedural-statements
    else 
      procedural-statements
    end-on]

Clauses

Clause Description

BEGIN

Marks the start of a transaction. Only INDEXED files are logged.

If the ON clause is included, only the specified objects are logged. If the LEAVING clause is included, all objects are part of the transaction EXCEPT those specified in the clause.

The LEAVING and ON clauses have no effect in a relational database version. All objects stored in the RDBMS that are processed during the transaction will form part of the transaction.

Any record locks that are held at the start of a transaction are released (unlocked) before the start of the transaction.

COMMIT

Specifies that the transaction was successful and to make all changes to the specified objects permanent. This ends the current transaction.

ROLLBACK

Specifies that the transaction was unsuccessful and to undo any changes made to the specified objects. This ends the current transaction.

The "current record" status of objects affected by the transaction cannot be guaranteed after a rollback.

FREE-LOCKS

Frees (releases) locks on records affected by the transaction, as they would normally be released if processed outside of a transaction.

This clause is obsolete as it only used in C-ISAM implementations.

ON ERROR

Statements to perform if an error occurred.

The ON ERROR clause of any I/O operation can have a list of error numbers specified (directly after the ON ERROR). The ON ERROR section is only performed if the error is one of those listed. If the error is not in the list, the program aborts with a full error detail screen.

Notes

  • BEGIN WORK, COMMIT WORK and ROLLBACK WORK are synonyms for TRANSACTION BEGIN, TRANSACTION COMMIT and TRANSACTION ROLLBACK respectively.

  • All record locks held are released (unlocked) when a transaction begins. Any record locks generated during the transaction are released when it is either committed, or rolled back.

  • All records inserted, updated and deleted during the transaction will remain locked until the transaction is completed (either committed or rolled back).

  • You should avoid transactions that are excessively long in either duration or records affected. Long running transactions can cause problems with other applications since all the records affected by the transaction are locked until its completion. If possible, locks to master files should be deferred until late in the transaction to limit the time that they remain locked.

  • Creating an object during a transaction will commit the transaction.

  • In most RDBMS implementations, insert, update and delete operations are more efficient when enclosed in a transaction. The process of beginning and committing a transaction does come with an overhead, however. Processing the same data with many small transactions is less efficient than processing it with fewer larger transactions.

  • Only objects with the INDEXED (ISAM) file type form part of the transaction. The DB-SEQUENTIAL file type also forms part of transactions on RDBMS systems. Other file types (for example, TEXT, SEQUENTIAL) do not form part of the transaction, and changes cannot be rolled back.
