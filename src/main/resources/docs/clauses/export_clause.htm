EXPORT clause

A Pronto RAD component library is an operations table designed to contain a group of routines that are directly callable but independent from a controlling program. A program can reference routines in multiple component libraries. A component library can reference routines in other component libraries.

Different declarations (routines), such as SCREEN, MENU, PROCEDURE, and API, can be marked with the EXPORT clause. Marking these routines with EXPORT in the component library program indicates that the routine is an entry point to an operations table when used as a component library.

The names of EXPORT routines must be unique between all component libraries attached to a program. That is, the name of an EXPORT routine cannot occur in different named component libraries.

The LINK statement can be used to attach a component library to a program. This statement specifies the names of the component libraries containing routines referenced by the program. Multiple LINK statements can be specified and can refer to an already specified component library. Component library names are specified in the same manner as standard Pronto Xi programs when these are called (for example, deb/m10sales). The full path names should not be specified.

Syntax

export|user-trigger [can-override|no-override]

Clauses

Clause Description

USER-TRIGGER

This clause can be used to define a trigger that will be run from Screen Customiser.

This clause should be used instead of the EXPORT clause when defining a trigger.

Only components defined with a USER-TRIGGER clause can be run as a trigger, however a USER-TRIGGER component can be referenced as a normal exported component.

CAN-OVERRIDE and NO-OVERRIDE

These clauses specify whether the routine is allowed to be overridden by custom component versions. If neither clause is included, CAN-OVERRIDE is assumed.

Notes

Each unique component library has its own set of data per attachment to the controlling program. A single component library is only attached to the controlling program once even if it is referenced by other component libraries.

When multiple versions of the same component library are referenced (that is, custom/override versions), each unique component library is attached separately.

If deb/m10sales is contained in both $CUS and $BMS, $CUS/deb/m10sales and $BMS/deb/m10sales are attached separately (each with their own data).

Data is not shared between the controlling program and any attached component library, other than by parameter passing and returning. Similarly, data is not shared between different component libraries. Fields of the same name referenced in the controlling program are independent to that of the same field referenced in any of the component libraries.

The same transaction control is applied across the controlling program and all attached component libraries. That is, there can only be one transaction active. Transaction commands issued will affect the controlling program and all other attached component libraries.

The controlling program and all attached component libraries share the same current working directory. Any change to the current working directory will affect the controlling program and all other attached component libraries.

The CLOSE ALL and UNLOCK ALL operations by default impact the controlling program and any attached component libraries. To only affect files in the current operations table the LOCAL clause should be specified with the ALL clause. That is, CLOSE ALL LOCAL and UNLOCK ALL LOCAL.

The SPL/CALL statement without the LEAVE-FILES-OPEN or EXTERNAL clauses will issue a CLOSE ALL that will affect the controlling program and any attached component libraries.

Override (custom) component libraries

It is possible to have a layering of custom component libraries that override the same EXPORT routine. This is achieved by having multiple directories specified in PROPATH that each contain the same named component library. This enables support for localisation as well as customer-specific customisation.

Only the EXPORT routines that are intended to be overridden need be defined in the custom version of a component library.

If an override version of an EXPORT routine exists, the base (distribution) version is not started unless the override version explicitly passes control to it through the DO NEXT statement. That is, unless an override of the routine intends to be a complete replacement for the routine, it must manually pass control onto the next level of the routine.

If a component library wants to directly call only the local version of an EXPORT routine (that is, the one defined in that component library), it must use the DO LOCAL statement. This option must be used with extreme care as you will potentially bypass essential logic in an overridden version of the routine.

Overriding standard programs

The component library mechanism can also be used to override specified routines in standard Pronto Xi programs that are designed to allow this.

This is achieved by specifying the EXPORT clause on the routines allowed to be overridden in the standard $BMS program.

The main routine must have EXPORT specified. These routines must be structured to work as standard component library entry points.

To override these routines, a custom program of the same name would be placed in $CUS. When called, the version of the program in $CUS would be executed as the controlling program and refer to the $BMS version as a component library. The override program would have a main routine that issues the DO NEXT main statement to pass control to the standard program’s main routine. Any EXPORT routine called in the $BMS version will perform the $CUS version of the routine (if present).
