PARAMETER declaration

The PARAMETER declaration is a clause of the SCREEN, MENU and PROCEDURE declarations. This identifies the fields that will contain the parameter values passed by the DO statement which calls the routine. If this is the initial (for example, MAIN) routine executed in the program, the fields will contain the parameter values passed to the program (for example, by the SPL/CALL statement).

This function has the same syntax as the FIELD declaration except that it is local in scope to the procedure, screen or menu in which it is defined. It must appear immediately after the screen, menu or procedure declaration and before any procedural code.

The data storage for a local field is allocated upon each invocation of the routine (and de-allocated upon its exit), allowing for true recursive routines. A local field can only be referenced in the routine that defined it.

No error/warning is issued if the name of a local field matches the name of a globally defined field. The local field will remain totally independent of any global field with the same name. All explicit references to the field in the routine will apply only to the local field.

Syntax

parameter[s] [is|are] parameter [parameter ...]
   [returning return-parameter [return-parameter ...]]

where
parameter = field-name field-definition
parameter = object-name.* [global]
parameter = row-name.* like object-name.*
return-parameter = field-name field-definition
return-parameter = row-name.*

Notes

Parameters for a routine can be defined as list of fields (first syntax above), or a complete row/record for an object. This allows the entire row to be passed without the need to specify individual fields. Rows can also be specified as return values.

The entire contents of the row/record are copied. When only a subset of the record is needed it can be more efficient to pass individual fields.

This mechanism passes row values to/from routines defined in component libraries.

Multiple row parameters can be passed and returned. Other parameter fields can also be passed and returned.

A previously defined row can be returned (that is, a row defined as a parameter). The values of a globally defined object are also allowed to be returned.

Parameter rows defined using the LIKE clause are automatically defined as NO-JOINS. This means that when referencing a field in a row this must be prefixed with its <row-name>.

message "deb.accountcode is ", deb.accountcode

Field values in a row can be copied using the same mechanism used for NO-JOINS objects. For more information, see OBJECT declaration.

The <row-name> row parameter is not an object definition. File I/O operations cannot be performed using <row-name>.

A routine can also receive a row into a globally defined object. When a row is received into a global object parameter the fields in that object can be referenced as normal, without the need to prefix the object name unless the object itself is defined as NO-JOINS.

Retrieving a row into a global object is similar to performing a RESTORE operation. The current row for the object is not affected. The physical file/table being processed by the object can be different to that of the object whose row was passed.

Routines that receive global object parameters should not be called recursively, as the same global data area is used for each call.

The GLOBAL clause is intended for passing rows to component libraries only and might be of no benefit if called in the same component library or operations table.
