OBJECT declaration

This statement defines new objects to be used by the program that are not defined in the dictionaries used.

This statement is non-procedural. It can occur anywhere in the program without affecting the program’s logic flow.

Objects declared using this declaration are global to the entire program; however, they cannot be referenced until after they are defined.

Syntax

object object-name
         [like object-name-1]
         [type [is] file-type]
         [separator [is] alpha-literal]
         [file [is] alpha-literal]
         [record [is]
           field-name [type [is] type]
                      [pic [is] picture]
                      occurs occurrence [times]
                      [like field-name-2]
           field-name
              .
              .
              .
          end-record]
         [key [is] field-name [descending|desc] ...
              [unique] [db-index-only] [compress]]
             .
             .
             .
         [no-joins]
         [no-rowid]
       object-name
        .
        .
        .

Clauses

Clause Description

LIKE

This indicates that object-name is an alias of object-name-1. object-name contains all the definitions of object-name-1 but is considered a separate object whenever referenced. The LIKE clause allows you to maintain two independent indexes into the file; there is still only one file, however.

If an OPEN TRUNCATE statement is executed on the like object, all the records in the original file are deleted. You can create a separate file by using the OPEN statement with the FILE IS clause to specify a separate disk file.

TYPE

Specifies the type of file the object refers to.

For more information, see File types.

SEPARATOR

Defines the separator character used for EXTERNAL type files.

The separator defined must be an alphanumeric literal string of one character in length.

FILE

Defines the path-name of the physical file for the object.

RECORD

Defines the record layout for the object.

The field definitions are the same as that used in the FIELD statement. If the definition of the required field is defined in the dictionary or in a previous FIELD statement, only the field name should be specified.

KEY

Defines the fields that make up a key when the object is indexed.

If the object has multiple keys, specify each key with a separate KEY clause.

DESCENDING or DESC

Descending fields can be defined in an index for an object.

If the object is defined with an OBJECT statement, descending key fields can be specified by placing the DESCENDING (or DESC) clause directly after the field’s name in the KEY IS clause.

Descending key fields retrieve their highest value first. Be aware of this when positioning the file with the POSITION statement.

DB-INDEX-ONLY

This clause must occur after the field list for the key being defined. The clause specifies that the index is only to be created on relational database versions. These indexes can only be defined after all normal indexes.

COMPRESS

Specifies that full key compression be used by this index (if supported).

This can reduce the storage requirements for the index on some systems, although it is likely to increase the processing overhead of index maintenance.

NO-JOINS

The NO-JOINS statement specifies to not join any fields defined for this object to other fields of the same name in the program.

With some exceptions, all references to fields from the defined object must be prefixed with the name of the object.

If the object being defined is called deb and it contains the field accountcode, you must refer to the field as deb.accountcode. The field deb.accountcode is considered a different field to accountcode.

In circumstances where only fields from the object can be specified (for example, ON INDEX), the field can optionally be specified without the object prefix.

The values of prefixed fields can be copied between NO-JOINS objects and also imported/exported to their non-prefixed counterparts with the following SET statements:

set *              = object-name.*
set object-name.*  = *
set object1-name.* = object2-name.*

NO-ROWID

This clause is primarily for RDBMS tables. It specifies that rows from the table cannot be retrieved by ROWID (for example, a multi-table view).

NO-ROWID objects cannot lock, insert, update or delete records. They also cannot be used as a PRIMARY object in a SCREEN. They can be used with other file I/O statements (for example, GET, SELECT and EXTRACT).

The fields for the object are specified individually to the RDBMS and never by a wildcard (*). This allows only a portion of the actual fields that make up the table to be selected.

File types

FILE TYPE Description

EXTERNAL

Variable length record. Fields are separated by the separator character.

A CR/LF terminates each record (on UNIX and Microsoft Windows).

TEXT

Variable length record (trailing spaces from the last field are removed).

No separators between fields. Each field has the length of the field itself. A CR/LF terminates each record on Microsoft Windows, while on UNIX the record terminator is a LF.

SEQUENTIAL

Same as TEXT except there are no separators between records. This is one long ASCII data stream.

FIXED-LENGTH-TEXT

Same as TEXT except the trailing spaces at the end of the last field in each record are not removed and the file has a CR/LF combination terminating each record (on Unix and Microsoft Windows).

INDEXED or

ISAM

Index-sequential file with one or more indices defined.

PRN / CSV

Text file with comma separators (separator clause is ignored if present).

Text fields are surrounded by quotes and the record terminates with a LF (on Unix and Windows).

DB-SEQUENTIAL

Equivalent to a sequential file in relational database systems.

The file is stored as a non-indexed table

MEMORY

The indices and a cache of records are stored in memory.

This is for temporary files only, as the records are lost when the program exits. Only the program that created the memory file can access the records of the memory file.

Memory files can be defined and accessed in exactly the same way as normal indexed/ISAM files.

A file name is required to be specified for memory files, however, a file of that name is not created. The file name is required to allow LIKE objects to refer to the same memory file in the program.

The operations in a memory file cannot currently be rolled back.

Memory files can be used for very large temporary tables provided there are sufficient resources to store them (both memory and temporary disk space). If you declare an object as type ISAM and open it with the TEMPORARY clause, the runtime automatically uses a memory file.
