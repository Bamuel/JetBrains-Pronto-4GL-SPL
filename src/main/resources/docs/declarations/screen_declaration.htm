SCREEN declaration

Defines a screen for displaying and accepting data.

This screen can be called from another screen, menu, or procedure. If this is the first procedure defined (or its name is main) the program will start at this screen.

This statement automatically ends any previous SCREEN, PROCEDURE, or MENU declaration.

The DETAIL section loops until it is backed out of or until an EXIT statement is encountered. Backing out of or exiting the first procedure terminates the program.

Syntax

screen screen-name
   [responsive]
   [row-selection-object [is] memory-object-name [no-truncate]]
   [window [@row,col to @row,col]
      [relative to @row,col]]
   [window-position [is] numeric-value]
   [title expression]
   [primary object-name
      [on index index]
      [same|different [field-list]]]
   [select select-clauses]
   [quick-link [on] object-name]
   [no-quick-links]
   [colour [is] colour [on colour]]
   [review colour [is] colour [on colour]]
   [allowed [is|are] mode [mode-name ...]]
   [no-prompt-for-search|prompt-for-search]
   [find-for-currency]
   [prompt @row,col]
   [local field field-name field-definition ...]
   [local mode mode-name mode-definition...]
   [data-grid]
   [review occurs n [by n]]
   [review-from-start|review-from-current|review-from-end|review-bottom-to-top]
   [no-review-row-separators]
   [no-review-column-separators]
   [no-review-separators]
   [start-on-current-record]
   [leave-parent-screen]
   [when condition]
   [parameters[are] field-name field-definition ...]
   [returning field-name field-definition ...]
   [stop-exit-all-key]
   [position-on-ok|no-ok-cancel]
   [stay-in-correct]
   [form-entry]
   [export|user-trigger [can-override|no-override]]
   [help-context [is] help-file-expression  context-id-expression]
   [SCREEN-STYLE[is] <numeric-expression>]
...[SCREEN-STYLE	
   [before 
      procedural-statements]
   [detail 
      procedural-statements]
   [after 
      procedural-statements]
[end-screen]

Clauses

Clause Description

RESPONSIVE

The RESPONSIVE clause specifies that a Pronto Xi web interface screen is responsive. Within the SCREEN declaration, the RESPONSIVE clause must be used to indicate that the screen is responsive, otherwise it is a fixed position (non-responsive) screen.

For more information, see RESPONSIVE clause.

ROW-SELECTION-OBJECT

The ROW-SELECTION-OBJECT clause, and ROW-SELECTION-COUNT() and ROW-SELECTION-CHANGED() functions, allow you to select (tag) records from a set (usually the primary object or select on the screen) and then perform some action with the tagged (or non tagged) records.

For example, you could use this for:

  • selecting a number of orders to process or release from backorder

  • tagging multiple lines in a bank reconciliation process.

ROW-SELECTION-COUNT() and ROW-SELECTION-CHANGED() allow you to display things when something happens or apply logic to a mode button. For example, the Process button is active only when the ROW-SELECTION-COUNT() is greater than zero.

As records are tagged, the tagged records are inserted into a temporary object. You can read the temporary object to perform an action based on the index stored in that table.

The ROW-SELECTION-OBJECT clause is relevant for the screen declaration section of a DATA-GRID screen. It nominates an object of type memory to record the rows selected by the user. The key for index 1 of the object must specify the fields to uniquely identify selected rows.

When a row is selected, a record is added to the nominated object. The fields of the object will be set to the current values of the selected row.

When a row is deselected, the record with the corresponding index 1 key for that row will be deleted.

Any existing content of the row selection object will be cleared when the screen begins.

The NO-TRUNCATE clause can be added to the ROW-SELECTION-OBJECT clause of a SCREEN. It should be specified after the object name.

This indicates that the row selection object should not be truncated and any existing entries in the object should be retained.

If there is no existing temporary file for the row selection object, then one will be created automatically.

The syntax is:

ROW-SELECTION-OBJECT <object-name> [NO-TRUNCATE]

For more information, see ROW-SELECTION-COUNT() and ROW-SELECTION-CHANGED().

WINDOW

The screen is defined as being a window over the current screen. When the screen returns, the window disappears automatically.

If no coordinates for the window are specified, the window occupies the entire screen.

If RELATIVE TO is specified, the window is displayed relative to these coordinates.

The row and column values can be arithmetic expressions, but neither should exceed 255 or errors will occur at runtime.

Windows can be defined as larger than 80X24 in size. Windows larger than the normal size cause the screen to change to its wide mode, where available.

An attempt to call a screen with a larger window than the defined maximum size for the terminal results in a fatal error. The MAX-SCREEN-COLUMNS and MAX-SCREEN-ROWS functions can be used to determine the maximum available screen size for the current terminal.

Decimal addressing is not supported for data grids. The row and column coordinates for each field in the data grid must be specified as a whole number.

Refer to the notes section below regarding changes to the implementation of the mode line from the Pronto Xi Dimensions release onwards.

WINDOW-POSITION

This clause, which only takes effect if the WINDOW clause is also specified, is used to specify the initial position of the window, or to display it in full screen mode.

The allowed values are:

-1

Displays the window in full screen mode

1 to 9

Sets the initial position of the current window in the container window, where 1 = top left, 2 = top centre, and so on as shown in the following grid:

For example, to open the window at the bottom centre of the container window:

window-position is 8

TITLE

The TITLE clause is available for any MENU/SCREEN/PROCEDURE with a window. This will set the title on the title bar of the window. It is processed once only upon entry to the routine.

PRIMARY

This defines the primary object that the screen is accessing. All record searching is performed using this object.

If this clause is not specified, no automatic searching, reviewing, correction, insertion or removal of records can take place in the screen.

The ON INDEX clause forces the object to be accessed only on this index. If not specified the screen will access the object using index 1. index can be specified by the index number (for example, 1) or by listing every field (in order) which makes up the key to the required index.

The SAME option specifies that when searching, look only for records with the same values as those fields listed. If no fields are listed, it defaults to the fields that make up the specified index. The fields listed must match either all or part of the leading fields in the key for the specified index.

The DIFFERENT option specifies that when searching, show only each different (unique) value of those fields listed.

Only the fields listed (as being different) are returned to the screen. No other fields in the PRIMARY are returned.

SELECT

An SQL SELECT statement can be specified instead of a PRIMARY clause.

For more information, see SELECT clause.

QUICK-LINK

If the SELECT statement for the screen has multiple objects in the FROM list, this clause can be used to specify which object quick links for that screen will be stored against. If this clause is not specified, the first object is used.

NO-QUICK-LINKS

The NO-QUICK-LINKS clause specifies that the Quick Links options will not be available for this screen.

COLOUR or COLOR

The COLOUR clause is valid for a SCREEN/MENU/PROCEDURE to specify the default window colours. If only a single colour is specified, it is assumed to be the background colour.

When 0 or no colour is specified, the default foreground or background colour set by the user in the colour settings of the Pronto Xi Client is used, for example:

colour is red on 0

Windows RGB colour values can be specified by using the RGB-TO-COLOUR() function.

This clause is only valid if the WINDOW clause is also specified (with the exception of the main window).

The REVIEW COLOUR clause can be used in a review screen to specify the background colour for the review area. If only a single colour is specified, it is assumed to be the background colour.

ALLOWED

This clause lists all the modes that are allowed with this screen when no modes are specified in the call to the screen. If this clause is not specified, the default screen modes of SEARCH, ENTRY, CORRECT and REMOVE are displayed.

The following rules/restrictions apply:

  • If only one mode is specified, the screen is placed in this mode automatically. Otherwise, the available modes are displayed and the user must select one.

  • You can define your own screen modes using the MODE statement.

  • The following predefined modes can be used: SEARCH, ENTRY, CORRECT, and REMOVE. SEARCH expands to all the valid searching options for the screen, that is, FIND, NEXT, PREV, NEXT-SCR, PREV-SCR.

  • REMOVE and SEARCH are only valid if a PRIMARY object or SELECT is specified.

  • If this screen is called from another screen and the parent screen is in show mode, this screen is also in show mode and the specified screen modes (including ENTRY and CORRECT) are ignored.

NO-PROMPT-FOR-SEARCH

When search is specified as an allowed mode, the available search modes are prompted for along with the other allowed modes. If this option is specified, those search modes that can be selected by means of function or arrow keys are not prompted for.

PROMPT-FOR-SEARCH

Search modes that are normally not shown (for example, Next, Prev, Next Scr and Prev Scr) are displayed on the mode list.

FIND-FOR-CURRENCY

Specifies that whenever an allowed mode for this screen that requires currency on a record is selected, the Find mode is to be performed automatically beforehand.

PROMPT

Specifies the position to display the mode prompts on the screen.

If not specified, the mode prompts are displayed at a default position (generally the second bottom line of the current screen or window).

LOCAL MODE

Specifies a local mode. A local mode is the same as a standard global mode, but can be referenced only by the screen in which it is defined.

A local mode can refer to local and parameter fields in any conditions or expressions. The local/parameter fields must be defined prior to referencing them.

Local modes are automatically added to the allowed mode list for the screen if they are not explicitly listed.

An ALWAYS-SHOW clause will be assumed for a local mode unless a MODE-WHEN clause is specified. For more information, see MODE-WHEN.

Any conflicting screen clauses (for example, WHEN) must not be specified immediately after local mode definitions otherwise they will be assumed to be part of the mode definition.

Multiple mode definitions can be specified:

local mode  mode1 correct process
	mode2 entry prompt "entry"
	mode3 prompt "Do stuff"

A local mode cannot be passed in the mode list of the DO statement.

screen my-screen
parameter is p1 pic x
local mode local-correct correct process mode-when p1 = 1
allowed search local-correct

DATA-GRID

This is an enhanced version of a REVIEW screen. A data-grid is generated with moveable button style column headings, allowing columns to be resized and repositioned.

Hidden data grid columns are supported.

The column headings used are defined by the TITLE clause associated with each ACCEPT or DISPLAY in the DETAIL section of the SCREEN.

The column coordinate of the ACCEPT and DISPLAY statements that form the columns of the data grid are used for ordering and identification purposes only, and do not represent the actual location of the column.

A default of 20 rows is used for the data grid in a non-section (non-responsive) screen. In a responsive screen, a default of 5 rows is used. For more information, see SCREEN-SECTION statement.

REVIEW

Indicates that this is a review screen.

The number of different primary records in the review screen can be specified using the OCCURS clause. If not specified, a default number is used. A single record in a review screen can be spread over multiple lines. The number of lines per record is defined by the BY clause following the OCCURS.

OCCURS 10 BY 2. Co-ordinates for the additional lines are relative to the base (first) row specified in the DETAIL section.

REVIEW-FROM-START

Show records from the start of the file/list onwards when re-displaying the list of review records.

REVIEW-FROM-CURRENT

Show the previous screen of records starting at, and including, the current record when re-displaying the list of review records. The current record is maintained.

REVIEW-FROM-END

Show records from the end of the file/list backwards when re-displaying the list of review records. Not supported on relational databases.

REVIEW-BOTTOM-TO-TOP

This clause will cause the records in a review screen to be display from the bottom of the review area up to the top of the review area, instead of from top to bottom.

This is not the same as the REVIEW-FROM-END clause as the REVIEW-BOTTOM-TO-TOP clause reads forward through the list of records and not backwards from the end of the records. It therefore reverses the orientation of the records on the screen.

The REVIEW-FROM-END clause can be simulated by a combination of this clause and a SELECT statement with an ORDER BY specifying a DESCENDING order.

NO-REVIEW-ROW-SEPARATORS

NO-REVIEW-COLUMN-SEPARATORS

NO-REVIEW-SEPARATORS

The drawing of separator lines between columns and rows in a review screen can be suppressed with the following clauses: NO-REVIEW-ROW-SEPARATORS, NO-REVIEW-COLUMN-SEPARATORS, and NO-REVIEW-SEPARATORS (the latter suppresses both).

This only works on review screens, not data grids.

START-ON-CURRENT-RECORD

If this clause is included and a PRIMARY SAME key is also specified in the SCREEN declaration, the file is not positioned at the start of the key range if the current record is in that key range already. This also applies to screens defined with a SELECT, where by default the first record shown is the first valid record in the sequence specified by the ORDER BY.

LEAVE-PARENT-SCREEN

This clause only has an affect if specified for the initial routine with the program (for example, screen main).

If the program is called directly from another Pronto RAD program using the CALL or SPL statement, the contents of the screen in the calling program is retained instead of being cleared.

If the initial routine has a WINDOW specified, this clause is invalid.

WHEN

When searching for records, allow only records that satisfy the specified condition. This clause is only valid if used with a PRIMARY clause.

For optimum performance, avoiding using an expression in the WHEN clause where possible. For example, the following expression should be evaluated before the SCREEN declaration:

service-call-master.customer-code = uppercase(strconcat(ws-param-001))

LOCAL FIELD

The LOCAL FIELD declaration defines fields that are local in scope to this routine.

For more information, see LOCAL FIELD declaration.

PARAMETERS

Allows the definition of parameters that are passed to the routine.

For more information, see PARAMETER declaration.

RETURNING

Allows the definition of parameters that are returned to the calling routine.

For more information, see RETURNING declaration.

STOP-EXIT-ALL-KEY

When F8 is pressed, the runtime automatically backs out all programs being run by the user.

It accomplishes this by flagging that all requests for keyboard input are to be treated as though the Esc key had been pressed, to try and allow the programs to back out gracefully. All screen output is stopped while in this mode.

The EXIT-ALL operation is cancelled if the program executes a BACK-TO-DETAIL statement, as the intention of this statement is to stop users from being able to exit the screen by pressing Esc.

The STOP-EXIT-ALL-KEY clause was added to the SCREEN, MENU and PROCEDURE declarations to also stop the EXIT-ALL operation. The automatic backout is cancelled when a routine with this clause specified is encountered.

When this clause is included in a routine, the EXIT-ALL operation is not stopped if the user presses the EXIT-ALL key while in the routine. It only takes effect if the EXIT-ALL key is pressed in a child routine.

Any code that loops to stop a user from being able to exit by pressing Esc (other than by using the BACK-TO-DETAIL statement) can cause the application to hang if the EXIT-ALL key is pressed. Any routines that contain this type of code should either be rewritten or specify the STOP-EXIT-ALL-KEY clause to avoid this problem.

As a fail-safe, a 10 second limit is placed on the EXIT-ALL operation. If the EXIT-ALL is still in effect after 10 seconds, it is automatically cancelled.

This should not be relied upon, as this timeout is not available on all systems.

POSITION-ON-OK

NO-OK-CANCEL

This clause is used for screens that have no primary clause (for example a report parameters screen).

The POSITION-ON-OK clause for a screen indicates that the OK button should be positioned on initially (that is, gain focus) when the screen is in Correct mode.

If the screen requires confirmation based on the CONFIRM statement, the Cancel button is the default instead of OK if the default for the CONFIRM is NO.

The runtime automatically displays OK/Cancel buttons when the screen is in entry or correct mode. They are not shown for entry in a procedure.

When the Entry or Correct mode is selected in a screen, the screen’s modes are hidden and the OK and Cancel buttons are displayed in the centre of the line used by the modes.

Clicking the OK button is equivalent to pressing F4 (Confirm). Clicking the Cancel button is equivalent to pressing Esc.

The OK/Cancel buttons only gain focus for keyboard entry if the screen would normally ask for confirmation for entry/correct. The OK/Cancel buttons replace the confirmation question in this situation.

A screen in correct mode that does not have a PRIMARY defined is assumed to be an options/defaults screen. In this situation the OK button initially has the keyboard focus, instead of the first ACCEPT.

Only the screen that defines the current window can gain keyboard focus on the OK/Cancel buttons. Sub-screens inheriting the window of the calling routine do not give focus to the OK/Cancel buttons. They can still be clicked when a sub-screen is being processed.

The OK/Cancel buttons are not shown on a window that is less than four lines deep.

The NO-OK-CANCEL clause can be used on a screen to disable this functionality. If the CONFIRM statement is used, the NO-OK-CANCEL clause changes the way the confirm is presented to the user when the screen is in Correct mode. For more information, see Notes.

STAY-IN-CORRECT

When the screen is in Correct mode and the user finishes correcting the screen, if this clause has been specified, instead of control passing back to mode selection the next valid row in the screen is read and the screen is placed back into correct mode for that row. This continues until the user cancels correct mode or there are no more valid rows.

FORM-ENTRY

If this clause is included, the Entry mode of the screen is processed differently to the traditional sequential Entry mode.

When Entry mode is selected, all fields with an associated ACCEPT statement are set to their default values. The screen then operates the same as in Correct mode, that is, the user can navigate to any field.

When a screen defined as FORM-ENTRY is setting the default field values at the start of Entry mode, the following expression evaluates to TRUE in any VALIDATIONS or ON CHANGE sections that are executed at that time. This enables you to write separate validation logic for when the user is entering a value (expression evaluates to FALSE) and the automatic setting of the default value (expression evaluates to TRUE).

if SCREEN-MODE() = FORM-ENTRY-DEFAULTS

On exiting Entry mode, the VALIDATION section of all ACCEPT fields are processed.

The screen area associated with a DISPLAY, ACCEPT, etc. with a WHEN clause are greyed if the WHEN condition evaluates to FALSE. The WHEN clauses for all statements in the screen are evaluated each time screen input is requested.

A FORM-ENTRY screen with an ACCEPT with FILL WITH TRAILING specified will always have the trailing character added to the current field value whenever the ACCEPT is processed, even in Correct mode.

The boolean function FORM-ENTRY-VALIDATION() indicates if the screen is performing the final form validation in a FORM-ENTRY or RESPONSIVE screen. For more information, see FORM-ENTRY-VALIDATION().

EXPORT

Indicates that the routine is an entry point to an operations table when used as a component library.

For more information, see EXPORT clause.

USER-TRIGGER

This clause is similar to the EXPORT clause described above, but also allows the routine to be called as a component library trigger by Screen Customiser.

CAN-OVERRIDE

NO-OVERRIDE

These clauses specify whether the routine is allowed to be overridden by custom component versions. If neither clause is included, CAN-OVERRIDE is assumed.

HELP-CONTEXT

Specifies a context-sensitive help topic to display when the user presses F1 while this screen is displayed.

help-file-expression is an alpha expression or literal that specifies the name of the help file. context-id-expression is a numeric expression or literal that specifies the context ID of the topic in that help file that is to display.

This setting will override the help context information stored in the System Modules & Functions (SYS M412) function for the menu function that was used to access this screen, or a help context defined using the SET-HELP-CONTEXT function.

This feature requires runtime version 7.1 or higher and a .chm, .hlp or .htm format context-sensitive help system that has been generated using a suitable help authoring tool.

BEFORE

Specifies the BEFORE command group.

This should contain any setup code for the screen (for example, headings)

This is performed once at the start of the screen and whenever the screen is refreshed using the REFRESH statement.

DETAIL

This contains the iterative code for the screen.

This code is performed for every screen operation (for example, entry, search, and so on).

This command group should contain ACCEPT statements for entering or specifying field values.

You must have at least one valid input field in a screen. If you want to effectively hide it, that field can be placed outside the window bounds.

AFTER

Specifies the AFTER command group.

This should contain any finalisation code for the screen.

This is performed once when exiting the screen.

SCREEN-STYLE

Specifies a numeric screen style which will be sent to the web client.

NO-MODES

Defines a screen that has no mode selection or field input.

The screen is closed by clicking on the X close button for the window or by pressing Esc.

The OPTION statement can be used to define clickable buttons within the screen.

An EVENTS-SECTION may be specified to process screen events.

If a PRIMARY or SELECT clause is specified then either the first row or first page of the result set will be displayed.

Specifying an EVENTS-SECTION with a TIMEOUT and issuing a REFRESH DATA or REFRESH REVIEW will continually update the data if a PRIMARY or SELECT was specified.

ACCEPT, RADIO-BUTTON, CHECK-BOX and CONFIRM statements are not valid in a NO-MODES screen.

Notes

  • There is a special case where the name of the screen is main. In this circumstance the procedure is the first performed for the program regardless of its location in the program.

    Absolute row coordinates are allowed in a REVIEW screen.

  • Row coordinates outside the review area can be specified in the DETAIL section of a review screen. Any ACCEPT or DISPLAY statement that specifies a row outside the review area will function as in a normal screen.

  • The row coordinates for the first ACCEPT or DISPLAY statement identifies the first row of the review area. As a consequence, the first ACCEPT or DISPLAY statement cannot have an expression for the row coordinate.

  • Parameters for a routine can be defined as a complete row/record for an object. For more information, see PARAMETER declaration.

  • From the Pronto Xi Dimensions release onwards, the screen mode line has been separated from the standard screen area and can be moved using Settings > General in the Pronto Xi Enterprise Client. When defining the window for a screen, the height of the screen must still be specified as though the modes are displayed as buttons on the bottom row. Note also that the mode area is associated with the window for a screen. When a screen inherits the window of a parent screen, the use of a mode area is solely determined by the parent screen that owns the window.

  • Modes are displayed as traditional buttons, and the mode area is not separated in the following situations:

    • The screen specifies the location of modes using PROMPT @<row>,<col>.

    • The screen has a single mode not requiring mode selection or the OK/Cancel buttons to be displayed.

    • Only windows for screens create a mode area. Windows for procedures or menus do not create mode areas.

    • The screen inherits a parent window that does not have a mode area.

  • Screen windows when created are positioned in the available area not covered by the docked menu pane. If the screen window does not fit in the available area, the menu pane is set to auto hide mode so that it does not overlap the screen window.
