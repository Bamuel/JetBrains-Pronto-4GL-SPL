FIELD declaration

This defines a data field that does not belong in any of the objects being referenced. This statement is non-procedural. It can occur anywhere in the program without affecting the program’s logic flow.

Fields defined using the FIELD statement are global to the program, however they cannot be referenced until after they are defined.

The name of the field must be a valid name and must not be defined in the dictionary being used.

If only the TYPE clause is included, a default PIC is allocated. If only the PIC clause is included, a default type is allocated. The TYPE and PIC clauses can be used together, but neither can be used in conjunction with the LIKE clause.

Multiple fields can be defined in the one FIELD statement.

Syntax

field field-name
       [type [is] type]
       [pic [is] picture]
       [occurs occurrence [times]]
       [like field-name-2]
       [uppercase|lowercase]
       [justify]
       [use-name-in-db]
       [db-column-name [is] alpha-literal]
       [drill-back [is] alpha-literal]
      field-name
         .
         .
         .

Clauses

Clause Description

TYPE

Defines the type of field.

For more information, see Data types and Field maximum sizes (Alpha / String / Varchar).

PICTURE or PIC

Defines the type and size of a field.

For example, x(30) defines an alphanumeric field of 30 characters.

The picture is in the format of a standard picture string (without numeric editing characters).

LIKE

Sets the picture, type and occurrence values of the field to the same values as that of field-name-2. If this clause is used, neither the TYPE or PIC clauses can be specified.

UPPERCASE

The ACCEPT statement maps any lowercase characters entered to uppercase.

LOWERCASE

The ACCEPT statement maps any uppercase characters entered to lowercase.

JUSTIFY

The ACCEPT statement right-justifies any characters entered.

USE-NAME-IN-DB

The field’s name is used as its column name if this field is contained in a table stored in an RDBMS.

DB-COLUMN-NAME

Specifies the column name for this field if it is used in an RDBMS table.

DRILL-BACK

Defines the program to start to perform a data drillback for this field when it is included in an XML report.

The parameter being passed must be a global field and have a % either side of the parameter.

drillback is testprog %field-name%

Local fields cannot be specified as the parameter (between the %).

Data types

  • NUMERIC or DECIMAL

    Default numeric format

    This is stored as a fixed precision computational / binary value.

  • MONEY

    Local currency numeric format

    Fields which store local currency amounts should use this data type.

    The Decimal Currency field in the Pronto Xi System Parameters screen of the Pronto Xi System Administration Utility (proadmin) specifies whether the local currency contains decimal places. For more information, see Pronto Xi System Parameters. The PRODECIMALCURRENCY environment variable if set to Y will override this control setting, and if set to N will specify that decimal currency is not used.

    If the local currency does not contain decimal places, then fields of type MONEY will be shown without decimal places, and include (comma) punctuation every three integer places.

    If the local currency contains decimal places, then fields of type MONEY will be displayed as normal numbers.

    This is equivalent to the NUMERIC data type in how the number is defined and stored. That is, it is stored as a fixed precision computational / binary value.

  • FLOATING

    Floating-point numeric format

    All type FLOATING numeric fields are stored as double precision values.

  • ALPHA

    Alphanumeric character string of a fixed size

    The full size of the field defined (including trailing spaces) is used by most functions.

  • STRING

    Alphanumeric character string

    Trailing blanks (spaces) in the value of a field defined as type STRING are ignored by certain functions, for example, the CONCAT() function.

    If you define a field with just type string, it will default to 40 characters. That is:

    ws-fld-name type string

  • VARCHAR

    Large alphanumeric character string (runtime version 7.1 or higher only)

    This data type is more efficient than the STRING data type for large text fields. The VARCHAR data type can be used to declare strings of up to 16000 characters. If you do not specify the size, a default size of 128 characters is used. Note that databases limit the maximum size of VARCHAR fields. Oracle, for example, has a 4000 character limit. A VARCHAR field is not guaranteed to use less storage in the database than an ALPHA field. This is database dependent.

    The storage for a VARCHAR field is not dynamically allocated (only BLOBs have their storage dynamically allocated). You must define the maximum size that you need, and that space is allocated regardless of whether you store anything in the field.

    VARCHAR fields differ from ALPHA fields in that the actual string length of a VARCHAR value is stored separately so the runtime does not need to process all characters of the string's defined length to determine the actual length of the stored value. For example, to compare an ALPHA of PIC X(1000) to the literal "A", the runtime must test all of the 999 trailing bytes to ensure they are all spaces. With a VARCHAR, however, the runtime only needs to compare the number of bytes in the separately stored string length (which for "A" is 1). This means that a VARCHAR field is more efficient than an ALPHA field when smaller strings are stored in a large field.

    Trailing blanks (spaces) in the value of a field defined as type VARCHAR are ignored by certain functions, for example, the CONCAT() function.

    Within Informix there are two types of VARCHAR (VARCHAR and LVARCHAR). A standard VARCHAR (VARCHAR) uses a single byte to store the size of the data. This means that the size values are between 0 and 255. A VARCHAR declared with a size greater than 255 must be defined as a long VARCHAR (LVARCHAR). The way long VARCHARs are stored and passed is potentially less efficient than a standard VARCHAR.

    If you require a field to be around the 250 to 270 character length, it is recommended that you use pic x(250) instead of pic x(256). This means that the field will be defined as a VARCHAR and not an LVARCHAR.

  • BLOB

    Binary large object (runtime version 7.1 or higher only)

    This data type is intended to store large text strings or binary data for use with functions that support them, in particular the XML functions. It is only currently supported for fields defined in a program, not dictionary fields, so BLOB data cannot be written to the database.

    A BLOB field cannot be defined as an array. A BLOB does not have any storage pre-allocated to it. BLOBS are stored in memory, and storage is allocated dynamically once a value is assigned to the field. Allocating an exceptionally large document to a BLOB may cause the program to run out of memory, which could cause a program crash. The amount of memory available for BLOB data is system-dependent. If a BLOB field is assigned to another BLOB field using the SET statement or using the PARAMETER or RETURNING clause of a DO statement, only an internal pointer to the BLOB data is passed. Both fields will initially refer to the same BLOB data.

    The runtime maintains a reference count for each allocated BLOB, and multiple fields can reference the same BLOB area. The data for a BLOB is automatically deallocated when its reference count reaches zero. The reference count for BLOB data in a locally defined field is automatically decremented when the routine exits. The FREE-BLOB function can be used to deallocate the memory for a BLOB field. The value of a BLOB field can also be written to and from a file using the WRITE-BLOB-TO-FILE and READ-BLOB-FROM-FILE functions respectively.

  • XML-HANDLE

    Used to store the document handle or node handle returned by an XML function.

  • DATE

    Date format

    Dates from January 1, 1900 through to 2079 can be stored.

  • TIME

    Time format (in a single day)

    Times from 00:00:00 to 23:59:59 can be stored.

    A time is undefined if it goes negative or if it goes beyond 24:00:00. You can only work with date-time fields if that is possible.

    For example, if you want to reduce the time within a loop and use:

    // where lf-run-time type time.
while lf-run-time > time-from-date-time(lp-gmt,true)
    set lf-run-time -= asme-hours * 3600
endwhile
// Note this code could make the time go negative, this will hang the program.

    Instead, you should use a date time field:

    //where lf-next-due-datetime type date-time
while lf-next-due-datetime > lp-gmt
    set lf-next-due-datetime -= asme-hours * 3600
endwhile

  • DATE-TIME

    Date and time format (used for time stamps).

    See the Notes section below.

  • LOCAL-DATE-TIME

    Local date and time format (time zones are not applied)

    If you are entering a specific time for a different time zone, then you should use a LOCAL-DATE-TIME field so that local time zones are not applied. However, if you do that and you need to compare date/times, then this will make sense only for date/times in the same time zone.

    The actual time zone is never stored for a DATE-TIME field. Instead, the date/time that is stored is first converted to GMT time (based on the local time zone). The local time zone applied takes into consideration daylight savings. When displayed, the local time zone is reapplied, so that the date/time relative to your time zone is displayed. Anyone in a different time zone will see the date/time with their time zone applied (for example, 09:00 AM in Melbourne is 07:00 AM in Perth).

    A sales order is entered at 09:00 AM in Melbourne, and another is entered at 09:00 AM in Perth, but the server is in Adelaide.

    If 09:00 AM is entered in Melbourne, it is converted to GMT when stored (around 23:00 PM the previous day). If 09:00 AM is entered in Perth, it is stored in GMT relative to them (so around 01:00 AM). Neither relate to 09:00 AM in Adelaide.

    If the field is defined as LOCAL-DATE-TIME, then no time zone is processed. If you enter it as 09:00 AM anywhere, then that is what is stored and displayed.

    The important aspect that you need to know is what 09:00 AM means. If it is simply the time that the sales order was entered, then there is no problem. If it is a delivery time in Adelaide, then there will be a problem if it was entered in Perth, as in Perth the local time zone will be applied when converting it to GMT.

  • BOOLEAN

    Numeric format can be used to store TRUE/FALSE values

  • COLOUR or COLOR

    Numeric format used to store a colour

    See the Notes section below.

  • OLE-INTERFACE

    Used to store an OLE interface ID

    This data type can only be specified for fields defined in a 4GL program. This data type is not valid for dictionary based fields.

Precision restrictions

The Pronto Xi runtime uses double precision floating point for all internal arithmetic. This suffers rounding errors once you go past 16 digits (integer + decimal). This is a floating point restriction over which Pronto Xi has no control.

procedure main
 // Assume this field is defined as 18 digits // being 12 significant digits and 6 decimals places. // Incrementing by 0.000001 might be a problem // depending on the number of significant // digits in the whole number.
set sys-money-value = 1234567890
 // 10 significant digits 
message "Value Before:" sys-money-value
set sys-money-value += 0.000001
message "After Incrementing:" sys-money-value
 // Yields correct result 1234567890.000001 //
set sys-money-value = 12345678900
 // But with 11 or 12 significant digits
message "Value Before :" sys-money-value
set sys-money-value += 0.000001
message "After Incrementing:" sys-money-value
 // Does not provide the correct result

Field maximum sizes (Alpha / String / Varchar)

The limit for both alphanumeric and string fields is 1024 characters.

The compiler will assume a VARCHAR field type if you:

  • don’t specify ‘type alpha’ or ‘type string’, and

  • set the size - x(nnnn) greater than 1024.

ALPHA and STRING field types are both fixed length character field types, with maximum size of 1024 characters.

An ALPHA field defined as ‘pic x(1024)’ set with a single letter A will have 1023 trailing spaces added to the field.

A STRING field defined as ‘pic x(1024) type string’ set with a single letter A will be:

  1. padded with trailing nulls by the runtime

  2. padded with spaces within the database.

It is recommended that you use the VARCHAR field type instead of the STRING field type.

A VARCHAR (variable size character field) field type contains a character count, and padding is not required. Its maximum size is larger (currently 16000 characters) in working storage (memory) fields, 4000 characters in a database. Since they have no padding overhead, the database might be able to store them more efficiently.

One downside of a VARCHAR field type is that they can be less efficient for smaller fields or fields used in indexes, although this also depends on the database used.

In summary, the database constraints recommend:

  • not to use a VARCHAR as an index (because fixed-length text is optimal)

  • maximum VARCHAR size of 4000 characters when used in a database.

Notes

  • The format of NUMERIC and FLOATING types when stored in a RDBMS is the native RDBMS numeric format. This DATE-TIME data type is intended to store time stamp values returned by the DATE-TIME, GMT and SYS-TIME functions. DATE-TIME values do not have any special formatting characteristics and are displayed as a number only. These values should be displayed using the DATE-FROM-DATE-TIME and TIME-FROM-DATE-TIME functions.

  • Fields of type DATE-TIME which are directly associated with an ACCEPT or DISPLAY statement are interpreted as being GMT based (not local time based).

    When entering into a field defined as DATE-TIME, it will be entered based on the local time zone, however, the date / time that is stored is first converted to GMT time (based on your local time zone). When it is displayed or printed, your local time zone is reapplied, so that you see the date / time relative to your time zone. Anyone in a different time zone will see the date / time with their time zone applied. For example, in Melbourne it will show as 09:00 AM, and in Perth it will show as 07:00 AM.

  • Fields of type TIME should not be used to store these values, as the value will be lost (due to overflowing) if the field is written to an EXTERNAL file. This can happen if the file is being exported or imported.

COLOUR can be used for defining colour fields.

field
   ws-back type colour
   ws-fore type colour

procedure main
  set ws-back = blue
  set ws-fore = red
  do deb-maint
end-procedure

screen deb-maint
  primary deb-master
  window @1,1 to @22,80
  colour is ws-back on ws-fore
